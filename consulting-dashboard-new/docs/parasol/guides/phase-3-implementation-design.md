# Phase 3: 実装設計（技術的実現の設計）

**Phase**: 3/4
**目的**: パラソルドメイン言語から物理実装への変換設計
**前提**: [Phase 2: 詳細設計](./phase-2-detailed-design.md) が完了していること
**次Phase**: [Phase 4: 品質保証](./phase-4-quality-assurance.md)

---

## 📋 Phase 3 概要

Phase 3では、Phase 2で定義したドメインモデルとユースケースを、**技術的に実装可能な形**に変換します。

### Phase 3の成果物

- データベース設計書
- ロバストネス図
- テスト仕様書

### 所要時間

- 標準: 1週間
- 小規模サービス: 3-5日
- 大規模サービス: 2週間

---

## Step 3.1: データベース設計

**目的**: パラソルドメイン言語から物理データベースへの変換

### 作業手順

#### 1. エンティティ → テーブル変換

```markdown
パラソルエンティティ: プロジェクト [Project] [PROJECT]
↓
物理テーブル: projects
```

**変換ルール**:
- エンティティ名（英語名）をsnake_caseの複数形にする
- 例: `Project` → `projects`、`TaskDecompositionRequest` → `task_decomposition_requests`

#### 2. 値オブジェクト → カラム変換

```markdown
値オブジェクト: プロジェクト期間 [ProjectDuration]
↓
カラム: start_date, end_date, duration_days
```

**変換ルール**:
- 値オブジェクトの各フィールドを個別のカラムに展開
- 複合値オブジェクトはJSON型カラムも検討

#### 3. 集約 → テーブル関係

- 集約ルートは主テーブル
- 包含エンティティは外部キーで関連付け
- 不変条件はCHECK制約やトリガーで実装

### 使用テンプレート

`templates/dx-database-design.md`

### データベース設計の原則

- **正規化**: 基本的に第3正規形を目指す
- **パフォーマンス**: 必要に応じて非正規化
- **集約境界**: 集約ごとにトランザクション境界を設定
- **監査**: 作成日時、更新日時、作成者、更新者を標準カラムとする

---

## Step 3.2: ロバストネス図設計

**目的**: ユースケースからオブジェクト設計への橋渡し

### 作業手順

#### 1. 3要素の特定

- **Boundary（境界）**: ユーザーインターフェース要素
  - ページ、フォーム、ダイアログなど
- **Control（制御）**: ビジネスロジック・処理
  - サービスクラス、ドメインサービスなど
- **Entity（実体）**: データ・概念オブジェクト
  - エンティティ、値オブジェクトなど

#### 2. ロバストネス図の作成

```
[ユーザー]
   ↓
[Boundary: プロジェクト作成フォーム]
   ↓
[Control: プロジェクト作成サービス]
   ↓
[Entity: プロジェクト]
```

### 使用テンプレート

`templates/dx-robustness-diagram.md`

### ロバストネス図の原則

- **Boundary**: UIから直接アクセス可能
- **Control**: Boundaryから呼び出され、Entityを操作
- **Entity**: Controlから操作される、UIには直接露出しない

## Step 3.3: テスト仕様設計

**目的**: 品質保証のための包括的テスト戦略

### 作業手順

#### 1. テストレベル定義

```markdown
### 単体テスト（ドメインロジック）
- エンティティの不変条件
- 値オブジェクトのバリデーション
- ドメインサービスのビジネスロジック

### 統合テスト（サービス間連携）
- 他サービスユースケース呼び出し
- データベーストランザクション
- 外部API連携

### E2Eテスト（ユースケース検証）
- ユーザーシナリオの完全な検証
- ページ遷移とデータ永続化の確認
- 非機能要件の検証
```

#### 2. テストケース設計

```markdown
## テストケース: [ユースケース名]

### 正常系
- [ ] 基本フロー: [期待される動作]
- [ ] 代替フロー: [別パターンの動作]

### 異常系
- [ ] バリデーションエラー: [エラー時の動作]
- [ ] ビジネスルール違反: [制約違反時の動作]

### 境界値
- [ ] 最小値: [最小入力での動作]
- [ ] 最大値: [最大入力での動作]
```

### 使用テンプレート

`templates/dx-test-definition.md`

### テスト設計の原則

- **カバレッジ**: ビジネスルールを100%カバー
- **独立性**: テストケースは相互に独立
- **再現性**: 同じ条件で同じ結果
- **自動化**: 可能な限り自動化
